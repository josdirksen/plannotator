# Commit list view (linear history rail)

Status: rough spec, iterating. Not part of PR #990 — its own branch/PR, stacked
on `feat/simple-git` (it builds on the panel-view toggle and diff-type seams).

## What

A third left-panel view for code review: the linear commit history of the
current branch, newest first — `git log --oneline` as a clickable list.

```
● a1b2c3d  fix stage button border        2h    (current)
○ cd79196  round-6 review fixes           5h
○ 42024ab  round-4 review fixes           1d
─── origin/main ───────────────────────────
○ 9bded34  chore: bump version            2d
```

Clicking a commit shows **that commit's own diff** (git-show style: the commit
vs its parent). Each click is independent — this is "walk the history one step
at a time," not a sliding review baseline. The since-base/"what would a PR
show" question stays the Git status view's job.

## UI

- Panel toggle becomes three options: `Git status | Commits | Tree`.
  (Text toggle already shipped; adding one segment. If three segments crowd
  the 256px header, fall back to `Git status | Commits | Tree` at the same
  10px size — measure first.)
- List rows: short SHA, subject (truncated), relative age. Author shown only
  when it differs from the repo user. Current selection marked.
- A divider row where the branch meets the base (`origin/main` via the same
  merge-base we already resolve) — everything above it is "yours."
- The sidebar is a pure commit list — it never becomes a file list. Clicking
  a commit opens that commit's full diff in the center dock as the ALL-FILES
  view (the existing AllFilesCodeView surface): every file in one scrolling
  document. No accordion, no drill-in, no back button. (Decided 2026-07-03.)
- Depth: `--first-parent -50`, "Show more" appends another 50. No graph, no
  topology, ever.

## Server

- New diff mode `commit:<sha>` (git only) along the existing seams:
  - patch: `git diff <sha>^ <sha>` (root commit: `git show` / empty-tree base)
  - label: `Commit <shortsha> — <subject>`
  - fingerprint: the sha itself — immutable, so the staleness probe is trivial
    (always fresh unless HEAD moved and the commit vanished from history)
  - file-content: old side `<sha>^`, new side `<sha>`
- New endpoint `GET /api/commits?limit=&before=` → `[{ sha, shortSha, subject,
  author, isRepoUser, ageRelative, isHead, isPastBase }]`. Computed against the
  same cwd/worktree as the active diff.
- `/api/diff/switch` accepts `commit:<sha>` as diffType. Sections sidecar: not
  produced (sections is since-base-only, unchanged).
- Staging gates OFF for commit diffs (nothing stageable in a historical
  commit). `isPathStageable` already centralizes this.
- Ask AI context: the agent-review message machine gets a commit-mode inspect
  string ("this is commit <sha>, inspect with `git show <sha>`").
- Both runtimes (Bun + Pi). jj/P4/workspace/PR mode: out of scope, toggle
  hidden (same `sectionsCapable`-style gate, git-local sessions only).

## Persistence

- The Commits view is SESSION-ONLY (revised 2026-07-03 after live review): it
  never persists as the opening view — a review always opens on the user's
  sections/tree default, and Commits is entered via the toggle each session.
  `reviewPanelView` stays `'sections' | 'tree'`; a stale `'commits'` cookie is
  treated as unset. No default-diff coupling either (unlike
  sections⟺since-base), and never persist a sha (it may not exist after a
  rebase).
